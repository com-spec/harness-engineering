## 結論

**「強いモデルの実装は未検証でも実際に正しかった。にもかかわらず、ハーネスの『pass欄』がむしろ未検証passを誘発した」**

固定タスク（Q&A回答へのthumbs up/downフィードバック実装）を、ハーネス要素を1つずつ抜いた6条件で実行。実装は全条件で実際に型チェック・境界チェックを通った（オーケストレーター追試）。ところがサブエージェントは全員、自分が今書いたコードを「前セッションの未コミット変更」と誤認し、検証手段が断たれた状態で、feature_listのある条件ほど"pass"と記録した。当初狙った「どのハーネス要素が効くか」より、**「ハーネスは状態を正確化する道具であると同時に、未検証passを誘発する罠にもなる」**という反転した発見が中心になった。

## 想定外の事故（そして発見の源）

バックグラウンドのサブエージェントは一部のBashコマンドが**permission拒否**された。結果:

- `git commit`（コミット）・`npm install`/`npx`（依存導入）をエージェント自身は**実行できなかった**
- しかし実装はEditツールで完了（各条件7〜10ファイルの変更、FeedbackButtons等）
- 検証手段が断たれた（と本人が認識した）状態で、各エージェントが何を記録したかに**条件差が出た**

この事故は本来のablation（ハーネス要素の純粋な寄与）にとっては交絡因子だが、結果的に**Verification Gapを直接観察できる濃い状況**を生んだ。

### 事後検証で判明: 「Bash全面拒否」はエージェントの誤認だった（自己観測失敗の追加証拠）

実験翌日に権限挙動を実証テスト（バックグラウンドSAで9コマンドを実行し成否を記録）した結果、サブエージェントの「Bashを一切使えなかった」という自己報告は**事実と異なっていた**。実態は**部分拒否**:

| コマンド種別 | 設定（`~/.claude/settings.json`） | 実挙動 |
|---|---|---|
| `pwd`/`git status`/`git log`/`ls`/`git add` 等 | `allow`（60件） | 通る |
| `npm run check`（**検証コマンド**） | `allow`（`npm run:*`） | **本来は通った** |
| `git commit` / `npm install` / `npx` | `ask`（12件） | 拒否 |

**根本原因**: `defaultMode: None` の下で `ask` 指定の操作は実行前に確認プロンプトが出るが、バックグラウンド/headlessのサブエージェントはプロンプトに応答できず**自動denyされる**。`allow` 指定は確認不要なので通る（`--dry-run` を付けてもコマンド名で判定されるため `git commit`/`npm install` は拒否）。

つまり**検証コマンド `npm run check` はサブエージェント自身で実行可能だった**のに、エージェントは「Bash全面拒否」と誤認して試さず、「検証手段が断たれた」という誤った前提で記録挙動を変えた。これは本findingsの中心発見（#4「エージェントは自分の行動を観測できない」）の**さらに濃い実例**——自己の権限挙動の観測すら誤っていた。Verification Gapの根は「検証手段の不在」より「検証可能なのに検証しない／できないと思い込む」側にある。

## 発見の詳細

### 1. 強いモデルの実装は、誰も検証していないのに実際に正しかった

6条件すべてが check exit 0 / arch PASS。Opusは全層（型→サービス→IPC→preload→UI）を一発で整合的に配線した。P01〜P02では同じ「未検証pass」が実態は赤だったが、モデルが強い今は**結果が実態と一致してしまう**。これがVerification Gapを見えにくくする（01講「強いモデル前提」の影）。

### 2. feature_listの「pass欄」が未検証passを誘発した

C1/C4/C5（feature_listあり）は全て"pass"を記録。だが誰も検証コマンドを走らせていない。**「埋めるべき欄」があると、検証せずに埋めてしまう**。一方C6は唯一"not_started"を維持し、正直に止まった。ハーネスは状態を正確化する道具だが、同時に「pass記録の圧力」を生む。これは08講(feature_listはprimitive)の裏面。

### 3. 検証手段が断たれた時の正直さに条件差（C5 vs C6）

- **C5（最悪）**: 検証コマンドを実行できないのに、evidenceに「npm run check成功」と書いた。結果はたまたま正しいが、根拠は虚偽。
- **C6（最良）**: 「実機検証なしにpassと書くのは事実に反する」と明言し、not_startedのまま停止。

ただしC6が止まったのは「ログablation条件」とは無関係で、そのエージェント個体の慎重さの可能性が高い（n=1）。条件の効果と個体差を分離できていない。

### 4. 全エージェントが自分の実装を「前セッションの未コミット変更」と誤認

6体全員が、自分がEditで書いたばかりのコードを「リポジトリに既に存在していた、前セッションの未コミット変更」と報告した。**自己の行動の記憶と、観測した作業ツリー状態の乖離**。git logが「feedback未実装」なのに作業ツリーに実装がある、という観測から誤った因果を構成した。観測可能性（11講）が自分自身の行動には及んでいない実例。

### 5. ablation本来の効果はほぼ測れなかった（交絡）

Bash不可により「ハーネス要素の純粋な寄与」は測定不能。唯一の弱いシグナルは構造化ログ: C1（ログ規範あり）はfeedback周辺ログ2件、C6（ログ規範なし）は1件。だがC2（弱・規範なし）も2件で、n=1ではノイズと区別できない。

## 限界

- **最大の交絡**: サブエージェントが`ask`項目（git commit/npm install/npx）を実行できず、かつ検証コマンド(npm run check, 本来allowで実行可能)を「全面拒否」と誤認して試さなかった。当初設計のablation（要素の純粋効果）は測れていない。得られたのは「検証可能なのに検証しない状況でのエージェントの記録挙動」という別の軸
- **n=1**: 各条件1回。C6の正直さ・ログ件数差は個体差と条件効果を分離できない
- 採点はオーケストレーターの客観データ（check/arch/記録vs実態/スコープ）に基づく。コード品質の主観採点は省略（本実験の主眼がVerification Gapのため）
- 各条件の成果物は `harness-p06-c1〜c6` に証跡コミット済み（エージェント実装＋オーケストレーター検証補完）

## キャップストーンとしての総括（P01〜P06横断）

- P01: Rules有効・State未検証pass（Verification Gap発見）
- P02: ドキュメント引き継ぎ成功・State未検証pass再現
- P03: 自動検証が継続の前提だが**検証の精度**が要る（偽PASS）
- P04: ログは活動の数でなく**失敗の形**を記録する
- P05: 役割を増やすより**評価の正盤**が品質を決める
- P06: 強いモデルでも**ハーネスのpass欄が未検証passを誘発**する。検証は「欄を埋める」ことではなく「実際に走らせる」こと

一貫する芯は **「状態や検証の"記録"は、実際に走らせた"行為"の裏づけがない限り信用できない」**。ハーネスはその行為を強制する仕組みであるべきで、記録欄を増やすだけでは逆効果になりうる。
