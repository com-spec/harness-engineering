## 結論

**「役割を増やすこと自体より、『何に対して評価するか』が品質を決める」**

同一機能要求（マルチターン会話履歴ビュー）を3体制で実装した結果、最終採点は single-role 3.75 / gen-eval 4.50 / plan-gen-eval 3.75 となり、コースの傾斜（1.6→3.3→4.9）は再現しなかった。特に plan-gen-eval では、計画書のスコープ判断が要求とズレた結果、ループ内Evaluatorは「契約準拠」で4.75を出しながら、要求基準のfresh採点では3.75に落ちた。役割分離は評価の基準（正盤）が要求と一致しているときにのみ機能する。

## 発見の詳細

### 1. コースの傾斜は再現しなかった——強いモデルは単独でも床が高い

single-role が 3.75（コースは1.6）。Opus は一人でも純粋関数分離・12テスト・型/境界クリアまでやり切り、コースが single-role の欠陥として挙げる「回答切り詰め・出典無視」は発生しなかった。01講「強いモデルを前提にせよ」の通り、モデルが強いとハーネス（役割分離）の効果幅は縮む。ただし消えはしない（gen-evalは+0.75）。

### 2. gen-eval の改善は「指摘→修正」の直接対応で実在した

ループ内Evaluatorの修正指示5件（日付つきタイムスタンプ・条件付き自動スクロール・フォーカス可視性・aria-live・大量履歴の明記）は、最終採点で評価された軸とそのまま重なった。external review が品質に変換される経路を確認できた。

### 3. plan-gen-eval の落とし穴: 契約が要求とズレると、多役割でも「ズレに最適化」される

- Plannerは堅実な判断として「変更は3ファイルに限定、App.tsx凍結」を契約に明記した
- その結果、フォローアップ質問の導線（要求3「続けて質問する」）が物理的に実装不能になった
- Generatorは契約に100%忠実（逸脱ゼロ）、ループ内Evaluatorも**契約の受け入れ基準で**検収して4.75で一発承認
- しかし要求基準のfresh採点では、まさにその欠落（フォローアップ・レイアウト）が減点され3.75
- **役割が増えるほど中間成果物（契約）への忠実度は上がるが、契約自体が要求とズレていると全員がズレへ向かって精密に働く**。P03の「検証の精度」、P04の「失敗の形を記録する」と同じ系譜で、「評価の正盤の精度」が品質の上限を決める

### 4. ループ内評価と独立採点の乖離（4.75 vs 3.75）

同じルーブリックでも「契約に対して」と「要求に対して」では1点差が出た。Evaluatorに何を正盤として渡すかが、レビューの意味を決める。

### 5. 実験の限界

- 採点はバリアントごとにn=1。0.75差はばらつきの範囲の可能性がある（採点者間の基準差も未統制）
- plan-gen-evalの結果は「Plannerの一判断」に大きく依存しており、計画の質が高ければコース通りの結果になった可能性がある
- 仕込み比較（コースsolution）は人間が書いた別実装であり、点数の直接比較は参考値

## ハーネスエンジニアリング概念との対応

| 発見 | 対応する概念 |
|---|---|
| external reviewの指摘が品質に直結 | Self-Verification / Evaluator分離（09-10講） |
| 契約へ精密に最適化されるが要求とズレる | 早期勝利宣言の変種: 「契約準拠」での勝利宣言 |
| ループ内評価4.75 vs fresh採点3.75 | 評価の正盤の精度（P03「検証の精度」の上位概念） |
| 強いモデルの単独floor が高い | 01講 強いモデル前提、ハーネス効果は差分で測る |
