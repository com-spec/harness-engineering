# Project 04 実験記録: Incremental Indexing（Runtime Feedback & Scope Control）

実施日: 2026-06-12
作業ディレクトリ: `~/Documents/GitHub/harness-p04-run1/`（コースリポ `projects/project-04/starter/` のコピー）
使用ツール: Claude Code（フェーズ1は前情報なしの general-purpose サブエージェント）

---

## 結論

**「観測性は『活動の数』ではなく『失敗の形』を記録する設計が要る」**

starterには仕込みバグ（1000文字超の文書で全チャンクが空になる）があり、既存ログは「チャンクを6個生成した」という活動量だけを記録していた。数のログは正常に見え、中身が空という失敗はログから見えない。調査エージェント自身が「チャンクの文字数や空チャンク検出のログがあれば即発見できた」と証言した。

---

## 実験デザインと経過

### フェーズ1: 前情報なしサブエージェントによるバグ調査（ログなし状態）

主エージェント（私）はコースページを読んで「大きいファイルのchunkingで失敗する」と知ってしまっていたため、調査は前情報なしのサブエージェントに委譲した。与えた情報はユーザー視点の症状のみ（「一部のファイルでインデックスが失敗する/完了しない」）。

結果: **6ステップ・約2分強・ツール13回で根本原因に到達**。

- 仮説A（主因）: `indexing-service.ts` の `content.length > 1000 ? '' : buffer.trim()` —— ロジックをNodeワンライナーで再現実験し「1940文字の文書→6チャンク全て空」を実測して確定
- 仮説B（副因）: 単一ドキュメントの `startIndexing(documentId)` が `index-meta.json` を更新せず、ステータスに永遠に反映されない —— コード追跡で確定
- 調査を遅らせた要因（本人証言）: ログがチャンク数しか出さない／「失敗」と「未完了」がUIで区別できない／自動テスト不在

**割引材料**: 仕込みバグには `// BUG:` コメントが明記されており、コードを読めば即わかる状態だった。実世界のバグはこのマーカーがないため、ランタイムログの価値は本実験の見かけ以上に大きいはず。

### フェーズ2: 観測性＋境界チェックの実装とバグ修正（TDD）

1. **テスト先行**: vitest で「1000文字超でも全チャンク非空」「単一ドキュメントでもstatus反映」の2件を作成 → **RED 2件**（バグと副因の両方を捕捉）
2. **修正**: バグ行の除去＋単一ドキュメント経路の `index-meta.json` 同期 → **GREEN 2/2**
3. **構造化ロガー** `src/services/logger.ts`: JSON Lines、コンソール＋ファイル永続化（`<dataDir>/logs/app.log`）。起動・import・indexing・IPCエラーの各経路に配線
4. **失敗の形を記録**: チャンク処理に総文字数・平均文字数の統計と**空チャンク検出のwarn**を追加（フェーズ1の証言に直接応えるログ）
5. **境界チェック** `scripts/check-architecture.sh`（`npm run check:arch`）: 10ルール（renderer/services/preload/mainの層間import禁止）＋ `docs/ARCHITECTURE.md`
6. **赤ベースラインの解消**: starter出荷時の型エラー14件（P02と同一パターン）＋ App.tsx の重複Window宣言（types.d.tsより弱い型で上書きしていた）を修正

検証: `npm run check` exit 0 / `npm test` 2/2 / `npm run check:arch` PASS / `npm run build` 成功。

## solution との比較

| 観点 | 自分の実装 | コース solution |
|---|---|---|
| バグ修正 | `buffer.trim()` に戻す | 同一アプローチ |
| ロガー | ファイル永続化あり、レベルフィルタなし | コンソールのみ、minLevelフィルタあり |
| 空チャンク警告 | あり（失敗シグナルを直接観測） | なし（チャンク数のみ） |
| 副因（index-meta未同期） | 修正済み | **未修正**（solutionでも単一indexはstatusに反映されない） |
| 境界チェック | 10ルール | 3ルール（うち「services/mainにReact禁止」は自分に欠けていたため採用） |

solutionも「数を記録する」ログに留まり、症状の片方（完了しない）は残っている。コースの参照実装すら観測性の設計が活動量寄りである点は、結論を裏づける。

## ハーネスエンジニアリング概念との対応

| 発見 | 対応する概念 |
|---|---|
| チャンク数のログでは空内容が見えない | Runtime Feedback の質（11講 観測可能性） |
| 空チャンクwarnの追加 | 失敗シグナルの直接観測（Evaluatorの入力になる） |
| check-architecture.sh による層境界の機械検証 | Scope Control の自動化（07講） |
| starterの赤ベースライン（3回連続） | Verification Gap（P01/P02/P04で再現） |
| `BUG:` コメント付きの仕込みバグ | 実験の限界: 実バグはマーカーがない |
