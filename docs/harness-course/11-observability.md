# Lecture 11: なぜ観測可能性はハーネスの内側にあるべきなのか

> コード例: [code/](https://github.com/walkinglabs/learn-harness-engineering/blob/main/docs/en/lectures/lecture-11-why-observability-belongs-inside-the-harness/code/) 実践プロジェクト: [Project 06. Build a Complete Agent Harness](https://walkinglabs.github.io/learn-harness-engineering/en/projects/project-06-runtime-observability-and-debugging/)

エージェントが20分走って大量のファイルに触れ、「完了。ただしテストが2件落ちている」と言う。理由を聞くと「不明、タイミング問題かも」。どのクリティカルパスを変えたか聞くと「コードを見てみます…」。

原因はエージェントの能力ではなくハーネスの観測可能性の欠如。実行時の実際の状態が見えないままタスクを実行すると、エージェントのすべての判断は本質的に推測になる。**観測可能性がなければ、エージェントは不確実性の下で意思決定し、評価は主観的判断になり、リトライは盲目的な放浪になる。** OpenAIもAnthropicも信頼性を「証拠の問題」として捉えている: ハーネスは実行時挙動と評価信号を、次の判断を実際に導ける形で露出しなければならない。

## 観測可能性の欠如の実コスト

- **「正しい」と「正しく見える」を区別できない**: コードレビューは「何が書かれたか」を示し、実行時トレースは「実際に何が走ったか」を示す。両方必要
- **評価が神秘主義になる**: 採点基準と受け入れ基準がなければ評価者は暗黙の前提に頼り、同じ出力に対する評価が評価者ごとにばらつき、再現不能になる
- **リトライが当てずっぽうになる**: なぜ失敗したか分からないままのリトライ方向はランダム。無関係なコードパスを直し続けてトークンと時間を燃やす
- **セッション引き継ぎの情報断崖**: 未完の作業を次セッションに渡すとき、観測可能性が欠けていると新セッションはシステム状態をゼロから診断する。Anthropicの観察では、この冗長診断がセッション総時間の30〜50%を食う

## Claude Codeの実シナリオ

planner-generator-evaluatorの3役ワークフローで「アプリにダークモードを追加」を実行した比較:

- **観測可能性なし**: plannerは曖昧な記述を出し、generatorはその曖昧さの上に実装し、evaluatorは自分の暗黙基準で却下するが「なんかしっくりこない」としか言えない。曖昧な却下理由に対する盲目的リトライが3〜4回繰り返され、約45分でようやく許容範囲の出力
- **完全な観測可能性あり**: plannerはスプリント契約（修正対象コンポーネント・各検証基準・除外事項）を出力。generatorは契約に従って実装し、実行時観測が各コンポーネントのスタイル適用過程を記録。evaluatorは採点ルーブリックで次元ごとに評価し具体的証拠を引用（「ボタンのコントラスト不足。WCAG AA基準4.5:1に対し実測2.1:1」）。1回のイテレーション・約15分で高品質な結果

効率差は3倍。唯一の変数が観測可能性。

## 2層の観測可能性

観測可能性は「ログを増やす」ことではない。2つの層で機能し、両方が必須。

- **実行時観測可能性（Runtime observability）**: ログ・トレース・プロセスイベント・ヘルスチェックなどシステムレベルの信号。「システムは何をしたか」に答える
- **プロセス観測可能性（Process observability）**: 計画・採点ルーブリック・受け入れ基準などハーネスの意思決定アーティファクトの可視性。「なぜこの変更を受け入れるべきか」に答える

## 核心概念

- **タスクトレース**: タスク開始から完了までの完全な意思決定経路の記録。分散システムのリクエストトレーシングに相当。問題発生時にプロセス全体を再生できる
- **スプリント契約（Sprint contract）**: コーディング開始前に交渉される短期合意。タスクスコープ・検証基準・除外事項を明記する。プロセス観測可能性の中核ツール
- **評価者ルーブリック（Evaluator rubric）**: 品質評価を主観的判断から証拠ベースの構造化スコアリングに変換し、異なる評価者が同じ出力に対して近い結論に至れるようにする
- **層状の観測可能性**: システム層とプロセス層を同時に設計し相互補強する。実行時信号は挙動を説明し、プロセスアーティファクトは意図を説明する

## なぜエージェント自身では解決できないのか

1. **エージェントは自分が知らないことを知らない**。必要だと気づいていない信号は記録しない。ハーネスレベルの制約がなければ、エージェントが重要と思うものだけがログに残り、それは通常足りない
2. **ログ形式が一貫しない**。セッションごとに形式が違い、系統的分析ができない
3. **プロセス観測可能性はログでは解決できない**。スプリント契約や採点ルーブリックは構造化アーティファクトであり、ハーネスレベルの支援が必要。print文を足しても代わりにならない

## 観測可能性の構築方法

1. **実行時信号の収集をハーネスに組み込む**: エージェント任せにせず、ハーネスが自動収集する。対象: アプリのライフサイクル（起動・ready・実行中・終了）、クリティカルパスの実行記録（入口・チェックポイント・出口）、コンポーネント間のデータフロー、異常なリソース使用パターン（メモリの単調増加など）、完全なエラーコンテキスト
2. **スプリント契約を実装する**: 各タスク開始前にgeneratorとevaluator（同一エージェントの別呼び出しでもよい）が、何を作り何をもって「完了」とするかを契約として合意する。スコープ・検証基準・除外事項（例: 印刷スタイルは扱わない）の3部構成
3. **評価者ルーブリックを確立する**: 「良いか悪いか」を定量スコアに変える。例: コード正確性（A: 全テスト合格 〜 D: ビルド失敗）、アーキテクチャ準拠、テストカバレッジの各次元をA〜Dで採点
4. **OpenTelemetryで標準化する**: ハーネスセッションごとにトレース、タスクごとにスパン、検証ステップごとにサブスパンを作成し、標準属性で注釈する。Jaeger・Zipkinなど標準ツールチェーンと統合できる

## Anthropicの3エージェントアーキテクチャ実験

2026年3月、Anthropicは「Web Audio APIでブラウザベースのDAWを作る」という同一タスクを複数アーキテクチャで実行し、フェーズごとの詳細データを公開した（総計3時間50分・$124.70。Planner 4.7分 → Build/QAを3ラウンド反復）。

- **Planner**: 1〜4文のユーザー要件を完全なプロダクト仕様に展開する。「スコープは大胆に」「詳細な技術実装ではなくプロダクト文脈と高レベル技術設計に集中せよ」と指示された。plannerが早期に細部の技術判断を誤るとエラーが下流に連鎖するため、成果物を制約し実行時の道筋はエージェントに探させる方がよい
- **Generator**: 機能ごと・スプリントごとに実装する。各スプリント前にevaluatorとスプリント契約を交渉し、その機能ブロックの「完了」の意味を定義してから実装・自己評価し、QAに引き渡す
- **Evaluator**: Playwright MCPで実ユーザーのように動作中のアプリを操作し、UI機能・APIエンドポイント・DB状態をテストする。プロダクト深度・機能性・ビジュアルデザイン・コード品質の4次元で採点し、どれか1つでもハード閾値を下回ればスプリントは不合格となり、generatorに詳細なフィードバックが渡る

QAラウンド1のフィードバック例: 「視覚的には印象的だがDAWのコア機能の多くが見かけだけで操作の深さがない: クリップをドラッグできない、シンセのノブやドラムパッドのUIパネルがない、EQカーブやコンプレッサーメーターのエフェクトエディタがない」。「しっくりこない」ではなく、具体的で証拠に基づくフィードバック。

evaluatorは最初から鋭かったわけではない。初期バージョンは妥当な問題を特定した後で「深刻ではない」と自分を説得して承認してしまっていた。修正方法は、evaluatorのログを読み、人間の判断と乖離した箇所を特定し、その具体的問題に対処するようQAプロンプトを更新すること。この開発ループを数回繰り返してevaluatorの採点は信頼できるものになった。

## 要点

- **観測可能性はハーネスのアーキテクチャ特性**。後付け機能ではなく、最初から設計に組み込む中核能力
- **2つの観測層は両方必須**。実行時信号が「何が起きたか」を、プロセスアーティファクトが「なぜそうしたか」を説明する
- **スプリント契約は整合を前倒しする**。予見可能な理由でevaluatorに即却下されるものをgeneratorが作るのを防ぐ
- **採点ルーブリックは評価を再現可能にする**。異なる評価者が同じ出力に近いスコアを出せる
- **観測可能性の欠如はセッション時間の30〜50%を冗長診断で浪費させる**
