# Lecture 06: なぜ初期化には専用フェーズが必要なのか

> コード例: [code/](https://github.com/walkinglabs/learn-harness-engineering/blob/main/docs/en/lectures/lecture-06-why-initialization-needs-its-own-phase/code/) 実践プロジェクト: [Project 03. Multi-session continuity](https://walkinglabs.github.io/learn-harness-engineering/en/projects/project-03-multi-session-continuity/)

新しいセッションで「検索機能を追加して」と頼むと、エージェントはすぐコードを書き始める。だが20分後にテストフレームワークの設定不備に気づき、さらにDBマイグレーションの形式も誤っていたと判明する。機能は最終的に完成するが、セッションの大半は「このプロジェクトがどう動くかの把握」に費やされる。

結論: 初期化（環境整備・検証コマンドの疎通・プロジェクト構造の理解）と機能実装は最適化目標が根本的に異なる作業であり、混ぜてはいけない。初期化は専用フェーズとして分離する。

## 2種類の根本的に異なる作業

- 実装フェーズの目標: 検証済み機能の量と質の最大化
- 初期化フェーズの目標: 以降のすべての実装の信頼性と効率の最大化

両者を混ぜるとエージェントは多目的最適化を強いられ、目に見える出力であるコード書きに自然と傾き、価値が次セッション以降にしか現れないインフラ整備が犠牲になる。

## 混ぜると何が起きるか

- **インフラが固まらない**: テスト設定が未検証、lintが緩い、進捗ファイルなし。欠陥は2セッション目に表面化する
- **未検証の蓄積**: テスト基盤が整う前に書いたコードは、後から設計欠陥が見つかり手戻りが増える
- **コンテキスト予算の浪費**: 初期化に予算を食われ機能は半分しか完成せず、次セッションもゼロから理解し直す「両損」になる
- **暗黙の前提の地雷**: 初期化中の判断（テストフレームワーク選定など）を記録しないと、次セッションが矛盾する選択をする（Vitestの上にJestを導入する等）

Anthropicの実験データでは、専用初期化フェーズを設けたプロジェクトはマルチセッション環境で機能完成率が31%高く、初期化への投資は次の3〜4セッションで完全に回収される。OpenAIのCodexガイドも「リポジトリは運用記録（operational record）」原則を強調し、初回から明確な運用構造を確立せよとしている。

## 核心概念

- **初期化フェーズ**: エージェントのライフサイクルの最初のフェーズ。後続実装の前提条件のみを確立し、機能開発は一切しない。成果物はインフラであってビジネスコードではない
- **起動準備チェックリスト**: 新しいセッションが曖昧さなく操作できる条件。「起動できる・テストできる・進捗が見える・次の作業を拾える」の4条件すべてが必須
- **ゼロから vs テンプレートから**: テンプレート開始はゼロ開始を大きく上回る
- **常時引き継ぎ可能（Always Ready to Hand Off）**: 口頭説明なしに、リポジトリの中身だけで作業を継続できる状態を常に保つ
- **最初のテスト合格までの時間**: 初期化効率の中核メトリクス
- **後続セッションの成功率**: 暗黙知に頼らずタスクを遂行できたセッションの割合。初期化品質の最良の尺度

## 初期化を正しく行う方法

最初のセッションは初期化のみを行い、以下を成果物とする。

1. **起動可能な環境**: 依存関係インストール済み、環境問題なし
2. **検証可能なテストフレームワーク**: サンプルテストが最低1件パスし、テスト基盤自体の正しさを証明する
3. **起動準備チェックリスト文書**: 起動・テスト・検証コマンド、現在の状態、プロジェクト構造を明記したドキュメント
4. **タスク分解**: プロジェクト全体を順序付きタスクリストに分割し、各タスクに明確な受け入れ基準（例: 「pytest tests/test_auth.py 全件パス」）を付ける
5. **gitコミットによるチェックポイント**: 初期化完了時にクリーンなチェックポイントをコミットし、以降の作業はそこから始める

補足原則:

- **テンプレートから始める**: 空ディレクトリから始めず、create-react-app 等で標準構造・依存・テスト基盤をプリセットする
- **初期化の完了基準**: 「どれだけコードを書いたか」ではなく、起動準備チェックリストの4条件（起動できる・テストできる・進捗が見える・次を拾える）をすべて満たすこと。受け入れチェックリスト例: `make setup` がゼロから成功する / `make test` で最低1件パス / 新セッションがリポの中身だけで「起動方法・テスト方法」に答えられる / タスク分解ファイルに3件以上のタスクがある / すべてgitにコミット済み

## 実例

Reactフロントエンドプロジェクトでの比較:

- **混合方式**: セッション1で足場作成と最初の機能実装を同時に実施。起動・テストコマンドの文書も進捗ファイルもタスク分解もなし。セッション2は構造の推測に約20分を消費
- **専用初期化方式**: セッション1は初期化のみ（テンプレートから構造作成、Vitest設定、サンプルテスト検証、チェックリストとタスク分解作成、初期チェックポイントをコミット）。セッション2の再構築時間は3分未満

プロジェクト全体では、混合方式の総再構築時間は専用初期化方式より約60%多かった。初期化に費やした20分は後続セッションで何倍にもなって回収される。

## 要点

- 初期化と実装は最適化目標が異なる。混ぜると両方の質が下がる
- 初期化の成果物はビジネスコードではなくインフラ: 起動可能な環境、検証可能なテスト、起動準備チェックリスト、タスク分解
- 初期化の検収は4条件で行う: 起動できる・テストできる・進捗が見える・次の作業を拾える
- ゼロからよりテンプレートから始める
- 初期化への投資は次の3〜4セッションで完全回収される。追加コストではなく先行投資である
