# Lecture 10: なぜエンドツーエンドテストは結果を変えるのか

> コード例: [code/](https://github.com/walkinglabs/learn-harness-engineering/blob/main/docs/en/lectures/lecture-10-why-end-to-end-testing-changes-results/code/) 実践プロジェクト: [Project 05. Let the agent verify its own work](https://walkinglabs.github.io/learn-harness-engineering/en/projects/project-05-grounded-qa-verification/)

Electronアプリにファイルエクスポート機能を追加するタスク。レンダラーコンポーネント・preloadスクリプト・サービス層をすべて実装し、各コンポーネントの単体テストは全部通過。エージェントは「完成」と言う。だが実際にエクスポートボタンを押すと、ファイルパス形式が誤っていて、プログレスバーは反応せず、大容量ファイルでメモリリークする。コンポーネント境界の欠陥が5件、単体テストはひとつも捕まえられなかった。

各部品は単体では「正しく」見えるが、つなぐと問題が出る。Googleのテストピラミッドが示す通り単体テストの厚い基盤は不可欠だが、そこで止まるとコンポーネント間の問題を系統的に見逃す。エージェントは最速のテストだけ走らせて完了宣言する傾向があるため、この問題はさらに深刻になる。**システムレベルの欠陥がないことを証明できるのはE2Eテストだけ。**

## 単体テストの盲点

単体テストの設計思想は隔離（依存をモックし対象ユニットに集中）。これが速度と精度をもたらす一方、系統的な盲点を作る。以下は実際につないで動かしたときにしか表面化しない。

- **インターフェース不一致**: レンダラーは相対パスを渡し、preloadは絶対パスを期待。双方の単体テストはモックで通過
- **状態伝播エラー**: マイグレーション後もORMキャッシュ層が旧スキーマのエントリを保持。毎回フレッシュなモック環境では露呈しない
- **リソースライフサイクル問題**: ファイルハンドル・DB接続・ソケットの取得と解放は複数コンポーネントにまたがる。テストごとに独立リソースを作って捨てる単体テストでは競合もリークも出ない
- **環境依存**: モック済みテスト環境では正しいが、実環境では設定差・遅延・サービス不可用で失敗

## E2Eテストは結果だけでなく行動を変える

見落とされがちな点: 自分の成果がE2Eテストで検証されると知っているエージェントは、コーディング行動そのものが変わる。

1. **コンポーネント間の相互作用を考慮する**: 単機能の孤立した視点でなく「このインターフェースは上流とどうつながるか」を問いながら書く
2. **アーキテクチャ境界を尊重する**: 境界制約のあるシステムで、E2Eテストが境界ルールの遵守を強制する
3. **エラーパスを処理する**: E2Eテストは通常失敗シナリオを含むため、例外処理を考えるようになる

OpenAIのCodexエンジニアリング実践の強調点: **エージェント向けエラーメッセージには修正指示を含める。**「Direct filesystem access in renderer」ではなく「Direct filesystem access in renderer. すべてのファイル操作はpreloadブリッジ経由にすること。この呼び出しを preload/file-ops.ts に移し window.api 経由で呼べ」と書く。これによりアーキテクチャルールが自動修正ループになる。

## 核心概念

- **コンポーネント境界欠陥**: AとBがそれぞれ単体テストを通過しても、相互作用が誤った挙動を生む。E2Eテストが最も得意とする欠陥カテゴリ
- **テスト十分性の勾配**: 単体テストで検出可能な欠陥 <= 統合テストで検出可能 <= E2Eテストで検出可能。層が上がるほど検出能力が増す
- **アーキテクチャ境界の強制ルール**: 「レンダラープロセスはファイルシステムに直接アクセスしない」のような設計文書上のルールを、実行可能な自動チェックに変える。「紙に書いてある」から「CIで動いている」へ
- **レビューフィードバックの昇格（Review Feedback Promotion）**: 繰り返されるコードレビュー指摘を自動テストに変換する。新カテゴリの再発問題を見つけるたびにルールを追加し、ハーネスは自動的に強くなる
- **エージェント向けエラーメッセージ**: 「何が悪いか」だけでなく「どう直すか」を伝え、テスト失敗を自己修正フィードバックループにする

## 実践方法

### 0. E2Eテストを書く前にアーキテクチャ境界を定義する

E2Eテストの前提はシステムに明確な境界があること。アーキテクチャが混沌としていれば、E2Eテストは「混沌全体が動く」ことしか証明できない。

OpenAIの経験則: **エージェント生成コードベースでは、アーキテクチャ制約は初日からの前提条件として確立すべき**で、チームが大きくなってから考えることではない。理由は単純で、エージェントはリポジトリの既存パターンを（不整合・劣悪なものでも）模倣するため、制約がなければセッションごとにドリフトが増える。

OpenAIは「レイヤードドメインアーキテクチャ」を採用: 各ビジネスドメインを Types -> Config -> Repo -> Service -> Runtime -> UI の固定レイヤーに分割し、依存は厳密に前方のみ。ドメイン横断は明示的なProvidersインターフェース経由。それ以外の依存はカスタムlintで機械的に禁止する。

鍵となる原則: **不変条件を強制し、実装を細かく管理しない（Enforce invariants; don't micromanage implementation）。**「データは境界でパースする」と要求しても、どのライブラリを使うかは指定しない。

### 1. ハーネスにE2E層を含める

検証フローに明記する: クロスコンポーネント変更を伴うタスクでは、E2Eテスト通過が完了の前提条件。Level 1 単体テスト（必須）/ Level 2 統合テスト（必須）/ Level 3 E2Eテスト（クロスコンポーネント変更時に必須）。必須レベルのスキップ = 未完了。

### 2. アーキテクチャルールを実行可能なチェックにする

すべてのアーキテクチャ制約に対応するテストかlintルールを用意する。例: `grep -r "require('fs')" src/renderer/ && exit 1` のような機械チェック。

### 3. エージェント向けエラーメッセージを設計する

失敗メッセージは3要素を含む: 何が悪いか（ERROR）、なぜか（WHY）、どう直すか（FIX）。

### 4. レビューフィードバック昇格プロセスを確立する

コードレビューで新カテゴリのエージェントエラーを発見するたびに自動チェックに変換する。1ヶ月後のハーネスは月初より大幅に強くなっている。

## 実例

Electronアプリのファイルエクスポート機能（レンダラーUI・preloadのファイルシステムプロキシ・サービス層のデータ変換が関与）。単体テスト段階では全コンポーネント通過（全依存モック）でエージェントは完了宣言。E2Eテストが明らかにした欠陥:

| 欠陥 | 内容 | 単体テスト | E2E |
| --- | --- | --- | --- |
| インターフェース不一致 | ファイルパス形式の不整合 | 見逃し | 検出 |
| 状態伝播 | エクスポート進捗がIPC経由でUIに返らない | 見逃し | 検出 |
| リソースリーク | 大容量エクスポートのハンドル未解放 | 見逃し | 検出 |
| 権限問題 | パッケージ環境での権限差異 | 見逃し | 検出 |
| エラー伝播 | サービス層の例外がUI層に届かない | 見逃し | 検出 |

5件すべてをE2Eテストが検出し、単体テストはゼロ。代償はテスト時間が2秒から15秒に増えたことだけで、エージェントワークフローでは完全に許容範囲。

## 要点

- **単体テストはコンポーネント境界欠陥に系統的に盲目**。隔離という設計こそが相互作用の問題を検出できない原因
- **E2Eテストは欠陥を検出するだけでなく、エージェントのコードの書き方を変える**。統合と境界に意識が向く
- **アーキテクチャルールは実行可能でなければならない**。文書で読まれるのを待つのではなく、コミットごとに自動チェックする
- **エラーメッセージはエージェント向けに設計する**。具体的な「直し方」を含め、自己修正ループを形成する
- **レビューフィードバックの昇格でハーネスは自動的に強くなる**。捕捉した欠陥カテゴリはすべて恒久的な防衛線になる
